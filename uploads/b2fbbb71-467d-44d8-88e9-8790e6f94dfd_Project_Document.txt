# Project Document: Architecture & Specifications

## 1. Introduction
This document serves as the technical blueprint for the Research Insights AI Platform backend reboot. It details the system architecture, data models, API contracts, and deployment strategies required to scale our Retrieval-Augmented Generation (RAG) capabilities.

## 2. System Architecture

The overarching architecture is a decoupled client-server model.
- **Frontend**: A lightweight, responsive web application built with HTML, CSS, and vanilla JS (or a lightweight framework), served statically.
- **Backend API**: A high-performance RESTful API built with Python's FastAPI framework.
- **Database Layer**: PostgreSQL serving as the primary relational store, with the `pgvector` extension operating as the vector database for storing and querying embeddings.
- **AI/LLM Layer**: Integration with external LLM providers (specifically OpenAI) for embedding generation and final insight synthesis.
- **Task Queue**: Background asynchronous workers managing the heavy lifting of document extraction and embedding generation.

## 3. Data Schema

The PostgreSQL database will consist of the following primary tables:

### 3.1 `documents`
Stores metadata about the uploaded files.
- `id`: UUID, Primary Key
- `filename`: VARCHAR, the original name of the uploaded file
- `file_path`: VARCHAR, internal storage path
- `file_size`: INTEGER, size in bytes
- `mime_type`: VARCHAR (e.g., application/pdf)
- `upload_date`: TIMESTAMP
- `status`: ENUM ('pending', 'processing', 'completed', 'failed')

### 3.2 `document_chunks`
Stores the extracted text chunks and their semantic embeddings.
- `id`: UUID, Primary Key
- `document_id`: UUID, Foreign Key to `documents.id`
- `chunk_index`: INTEGER, the sequential order of the chunk in the document
- `content`: TEXT, the actual text string extracted
- `embedding`: VECTOR(1536), the embedding vector generated by OpenAI (dimension size depends on the specific model used)

### 3.3 `insight_history`
Tracks all generated insights.
- `id`: UUID, Primary Key
- `query_text`: VARCHAR, the user's research question
- `format_type`: VARCHAR (e.g., 'detailed', 'simplified')
- `response_content`: JSONB, the structured output generated by the AI
- `created_at`: TIMESTAMP
- `document_ids`: ARRAY of UUIDs, tracking which documents were selected for this context

## 4. Core Workflows

### 4.1 Document Ingestion Flow
1. Client POSTs a multipart/form-data request containing the file to `/api/documents/upload`.
2. FastAPI validates the file type, saves the file to the local disk, and creates a `documents` record with status 'pending'.
3. A background task is queued. The API immediately returns a 202 Accepted response.
4. The background task reads the file, extracts text using layout-aware parsers, and splits the text into sliding-window chunks.
5. Emdeddings are generated in batches via the OpenAI API.
6. The chunks and vectors are bulk-inserted into the `document_chunks` table.
7. The `documents` record status is updated to 'completed'.

### 4.2 Insight Generation Flow (RAG)
1. Client POSTs a query and a list of selected `document_id`s to `/api/insights/generate`.
2. The user's query is converted into an embedding vector.
3. The backend executes a vector similarity search (cosine distance) against the `document_chunks` table, filtering strictly by the provided `document_id`s.
4. The top-K most relevant chunks are retrieved.
5. The LangGraph agent takes over, formatting a strict prompt containing the system instructions, the retrieved chunk context, and the user's query.
6. The LLM generates the response in a structured JSON schema.
7. The result is saved to `insight_history` and returned to the client.

## 5. Non-Functional Requirements
- **Scalability**: The system must handle up to 100 concurrent queries and 50 concurrent document uploads without crashing.
- **Latency**: Sub-second response times for historical insight retrieval. Under 10 seconds for insight generation (LLM dependencies notwithstanding).
- **Security**: SQL injection prevention via SQLAlchemy ORM. Strict input validation using Pydantic models. Cross-Origin Resource Sharing (CORS) configured strictly for the designated frontend domain.

## 6. Integration Testing Strategy
We will employ a robust testing suite utilizing `pytest`.
- **Unit Tests**: Over 90% coverage for the chunking logic, text extraction utilities, and LangGraph workflow orchestration.
- **Integration Tests**: Creating an ephemeral PostgreSQL instance in a Docker container during CI/CD to verify `pgvector` indexing and retrieval accurately.
- **Mocking**: All OpenAI API calls must be mocked in the test suite to prevent unnecessary billing charges and ensure deterministic test runs.
